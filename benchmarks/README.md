# Бенчмарк качества

Воспроизводимая оценка качества Researcher-агента системы.

## Что измеряется

**Recall@10** — доля известных landmark РКИ (эталонные публикации, считающиеся "obligatory reading" по теме), которые были найдены в top-10 источниках Researcher.

Если ассистент **не нашёл** landmark trial, который врач должен был бы увидеть — это пропуск важного доказательства. Эта метрика напрямую отражает клиническую полезность.

## Эталонные кейсы (`cases.json`)

5 клинических вопросов из 5 разных областей онкологии:

| Case | Тема | Landmark trial |
|---|---|---|
| 01 | NSCLC, PD-L1 ≥50%, pembro vs chemo | KEYNOTE-024 |
| 02 | EGFR+ NSCLC, osimertinib first-line | FLAURA |
| 03 | HER2+ MBC, T-DXd vs T-DM1 | DESTINY-Breast03 |
| 04 | Меланома, nivo+ipi vs ipi | CheckMate 067 |
| 05 | BRCA+ рецидив рака яичников, olaparib поддержка | SOLO2 |

## Запуск

```bash
python -m benchmarks.run_benchmark
```

Прогон ~1.5 минуты (5 кейсов × ~15 секунд). Результаты записываются в `results.md` и `results.json`.

## Текущие результаты

См. `results.md`. На версии v1.0 — средний Recall@10 = **0.60** (3 кейса с полным попаданием, 2 частичных, 1 пропуск).

## Анализ пропусков

**Case 03 (DESTINY-Breast03 не найден)**: вероятная причина — Verifier не генерирует синонимы лекарственных названий (T-DXd, DS-8201, Enhertu). PubMed индексирует публикации под брендовыми именами, и поиск по `trastuzumab deruxtecan` может не возвращать те же результаты, что `T-DXd`.

**Case 04, 05 (частичный recall)**: PubMed возвращает top-N результатов по релевантности; оригинальная публикация landmark trial может уступать по релевантности более поздним подгрупповым анализам с тем же ключевым словом.

**Возможные улучшения**:
- Расширить Verifier генерацией синонимов (бренд + INN + код разработчика)
- Увеличить per_query_limit в Researcher
- Использовать MeSH-термины вместо free text
